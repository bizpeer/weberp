# HR FLOW 개발문서

## 1. 프로젝트 개요

이 프로젝트는 사내 인사 운영을 위한 Firebase 기반 HR 관리 시스템이다. 주요 목적은 다음 기능을 하나의 웹 애플리케이션으로 통합하는 것이다.

- 출퇴근 기록 및 월간 근태 조회
- 연차/반차/병가/기타 휴가 신청 및 승인
- 경비결의서 작성, 증빙 업로드, 승인
- 공지사항 게시 및 열람
- 조직도, 부서/팀, 직원 계정 관리
- 급여 정보 입력 및 예상 실수령액 계산
- 관리자용 도메인/보안 설정

프런트엔드는 React + Vite + TypeScript로 작성되어 있고, 데이터 저장과 인증은 Firebase를 중심으로 구성되어 있다. 일부 관리자 기능은 Firebase Cloud Functions와 별도 Express 서버를 통해 보완된다.

## 2. 저장소 구조

```text
HR management/
├─ client/        # React 프런트엔드
├─ server/        # Express 기반 관리자 보조 서버
├─ functions/     # Firebase Cloud Functions
├─ firebase.json  # Firebase Hosting / Functions / Rules 설정
├─ firestore.rules
├─ storage.rules
└─ package.json   # 루트 실행 스크립트
```

### 2.1 디렉터리별 역할

#### `client/`

실제 사용자 화면이 있는 메인 애플리케이션이다.

- 라우팅: `client/src/App.tsx`
- Firebase 초기화: `client/src/firebase.ts`
- 인증 상태 관리: `client/src/store/authStore.ts`
- 공통 컴포넌트: `client/src/components/*`
- 기능 페이지: `client/src/pages/*`

#### `server/`

Firebase Admin SDK를 사용해 클라이언트에서 직접 처리하기 어려운 관리자 작업을 수행하는 Express 서버다.

- 진입점: `server/index.ts`
- Admin SDK 초기화: `server/firebaseAdmin.ts`
- 초기 관리자/문서 시드: `server/db.ts`

현재 확인된 주요 API는 비밀번호 초기화용 `POST /api/reset-password` 이다.

#### `functions/`

Firebase Callable Function을 제공한다.

- 진입점: `functions/index.js`
- 현재 구현 함수: `adminResetPassword`

이 함수는 관리자 권한 확인 후 특정 사용자의 Firebase Auth 비밀번호를 강제로 초기화한다.

## 4. 실행 및 배포 방법

## 4.1 루트 스크립트

루트 `package.json` 기준 주요 명령어는 다음과 같다.

- `npm run client:dev`
- `npm run client:build`
- `npm run server:dev`
- `npm run emulators`
- `npm run deploy`

### 4.2 프런트엔드 실행

```bash
npm --prefix client install
npm run client:dev
```

기본 개발 서버는 Vite이며 로컬 주소는 보통 `http://localhost:5173` 이다.

### 4.3 관리자 서버 실행

```bash
npm --prefix server install
npm run server:dev
```

기본 포트는 `3001` 이며 헬스체크는 `GET /health` 이다.

### 4.4 Cloud Functions 실행/배포

```bash
npm --prefix functions install
npm --prefix functions run serve
```

배포 시:

```bash
npm run deploy
```

이 명령은 프런트엔드 빌드 후 Firebase Hosting 및 Functions 리소스를 배포한다.

클라이언트는 `client/src/firebase.ts` 에서 다음 Vite 환경변수를 사용한다.

- `VITE_FIREBASE_API_KEY`
- `VITE_FIREBASE_AUTH_DOMAIN`
- `VITE_FIREBASE_PROJECT_ID`
- `VITE_FIREBASE_STORAGE_BUCKET`
- `VITE_FIREBASE_MESSAGING_SENDER_ID`
- `VITE_FIREBASE_APP_ID`
- `VITE_FIREBASE_DATABASE_ID`

서버는 `server/firebaseAdmin.ts` 에서 다음 값을 참조한다.

- `GOOGLE_APPLICATION_CREDENTIALS`

없을 경우 `server/serviceAccount.json` 을 기본 경로로 사용한다.

### 운영상 주의

- `server/serviceAccount.json` 은 민감한 서비스 계정 키 파일이다.
- 실제 운영 저장소에서는 커밋 제외가 권장된다.
- Firebase Hosting은 `client/dist` 를 서빙한다.
- Firestore 및 Storage 접근은 rules 파일로 제어된다.

## 6.2 인증/인가 흐름

인증 상태의 중심은 `useAuthStore` (`client/src/store/authStore.ts`) 이다.

주요 책임:

- Firebase Auth 로그인 상태 구독
- Firestore `UserProfile` 문서 조회
- 시스템 기본 이메일 도메인 구독
- 사용자 역할 및 추가 프로필 정보 관리
- 최초 비밀번호 변경 강제
- 임시 계정 문서를 실제 UID 문서로 마이그레이션

### 로그인 흐름

1. 사용자가 로그인 모달에서 ID 또는 이메일 입력
2. 입력값에 `@`가 없으면 `systemDomain` 기준 이메일 조합 시도
3. `UserProfile.email` 검색으로 실제 이메일을 찾는 보정 로직 수행
4. `signInWithEmailAndPassword` 시도
5. 실패 시 자동 계정 활성화 로직 수행
6. Auth 성공 후 `authStore`가 `UserProfile/{uid}` 문서 로딩

### 최초 계정 생성/활성화 흐름

조직 관리 화면에서 직원 생성 시 `temp_<timestamp>` 형식의 임시 Firestore 문서가 `UserProfile` 에 생성된다.

실제 직원이 처음 로그인하면:

1. Firebase Auth 계정이 생성되거나 기존 계정과 연결됨
2. `authStore.initAuth()` 가 이메일 기반으로 임시 문서를 검색
3. 임시 문서를 실제 UID 문서로 복제
4. `mustChangePassword: true` 설정
5. 임시 문서 삭제 시도

즉, 관리자 등록과 실사용자 Auth 계정 생성이 분리된 구조다.

### 권한 체계

코드상 역할은 다음 세 가지다.

- `ADMIN`
- `SUB_ADMIN`
- `EMPLOYEE`

추가로 이메일이 `bizpeer@` 로 시작하면 코드에서 마스터 관리자처럼 취급한다.

#### 화면 접근 기준

- `ProtectedRoute requireAdmin`
  - `ADMIN`, `SUB_ADMIN`, 마스터 관리자 허용
- `ProtectedRoute requireMasterAdmin`
  - `ADMIN`, 마스터 관리자 허용
- 비로그인 상태
  - 로그인 모달을 열고 루트로 리다이렉트

## 6.3 데이터 흐름

대부분의 업무 기능은 프런트엔드에서 Firestore/Storage를 직접 읽고 쓴다.

- 조회: `onSnapshot`, `query`, `where`, `orderBy`
- 등록: `addDoc`, `setDoc`
- 수정: `updateDoc`
- 삭제: `deleteDoc`
- 파일 업로드: Firebase Storage `uploadBytes`, `getDownloadURL`

예외적으로 민감한 비밀번호 초기화는 Cloud Functions 또는 Express Admin 서버를 사용한다.

## 8. Firestore 보안 규칙 요약

`firestore.rules` 기준 핵심 정책은 다음과 같다.

- 인증 사용자만 대부분의 데이터 읽기 가능
- `UserProfile` 기반 `role` 로 권한 판단
- `bizpeer@*` 이메일은 마스터 관리자 취급
- `divisions`, `teams`, `config` 쓰기는 관리자 이상
- `expenses`, `leaves` 는 본인 신청 가능
- `expenses`, `leaves` 승인 상태 변경은 관리자 또는 해당 본부의 `SUB_ADMIN`
- `attendance` 생성은 본인만 가능
- `notices` 작성은 관리자/부관리자 가능

### 중요 포인트

- 직원 본인은 `UserProfile` 수정이 가능하지만 급여 관련 필드는 수정 불가하도록 제한돼 있다.
- `config/system` 읽기는 `allow read: if true;` 로 열려 있다.
  - 로그인 전에도 기본 도메인 표시가 필요하기 때문으로 보인다.

## 10. 비밀번호 관리 구조

비밀번호 관련 경로가 세 군데 존재한다.

### 10.1 클라이언트 직접 변경

- 사용자 본인 비밀번호 변경
- `firebase/auth` 의 `updatePassword`

### 10.2 Cloud Functions 관리자 초기화

- `functions/index.js`
- 함수명: `adminResetPassword`
- `EmployeeManagement.tsx` 에서 `httpsCallable` 로 호출

### 10.3 Express 서버 비밀번호 초기화 API

- `server/index.ts`
- `POST /api/reset-password`

현재 코드 기준으로는 클라이언트 쪽 직원관리 화면이 Cloud Functions 방식을 사용하고 있으며, Express API는 별도 관리자 운영용 또는 이전 방식으로 보인다.

## 11. 서버/함수 상세

## 11.1 Express 서버 (`server/`)

주요 역할:

- 헬스체크
- Firebase Admin SDK 기반 비밀번호 초기화 API 제공

초기화 방식:

- `firebaseAdmin.ts` 에서 Admin SDK 시작
- Firestore 데이터베이스 ID로 `weberp` 사용

### `db.ts`

`initDB()` 함수가 존재하며 역할은 다음과 같다.

- `config/system.defaultDomain` 조회
- 마스터 관리자 계정 `bizpeer@{domain}` 준비
- 기본 UID `bizpeer`
- Auth 사용자 생성 보정
- `UserProfile/bizpeer` 생성

다만 현재 `index.ts` 에서 `initDB()` 호출이 보이지 않아 자동 시드가 실제로 수행되는지는 별도 확인이 필요하다.

## 11.2 Cloud Functions (`functions/`)

`adminResetPassword` 동작:

1. 호출자 인증 확인
2. 입력값 `uid`, `password` 검증
3. 호출자의 `UserProfile.role` 이 `ADMIN` 인지 확인
4. `admin.auth().updateUser()` 로 비밀번호 초기화

주의점:

- 마스터 관리자 판정(`bizpeer@*`)은 여기 반영되지 않고 `role === "ADMIN"` 만 검사한다.

## 12. UI/상태 관리 방식

### 전역 상태

전역 상태는 대부분 Zustand store 하나(`authStore`)에 집중돼 있다.

보관 상태:

- `user`
- `userData`
- `systemDomain`
- `loading`
- `isLoginModalOpen`
- `isManualChangeMode`

### 로컬 상태

각 페이지는 자체적으로 `useState` + `onSnapshot` 조합을 사용한다. 별도 API 계층보다 화면 컴포넌트가 Firestore에 직접 접근하는 방식이 많다.

장점:

- 구현이 빠름
- 실시간 반영이 쉬움

단점:

- 페이지별 로직이 커짐
- 중복 쿼리/정렬/권한 보정 로직이 분산됨
- 테스트가 어려움

## 13. 배포 구조

`firebase.json` 기준:

- Hosting public 경로: `client/dist`
- 모든 경로 `index.html` 로 rewrite
- Firestore Rules 연결
- Storage Rules 연결
- Functions 소스: `functions/`

즉, 정적 SPA는 Hosting에 배포되고, 서버리스 비밀번호 초기화는 Firebase Functions가 담당한다. 별도 `server/` 는 Firebase 배포 대상이 아니므로, 사용 시 별도 서버 환경이 필요하다.

## 14. 현재 코드 기준 이슈 및 주의사항

## 14.1 인코딩 깨짐 흔적

여러 파일에서 한글 주석과 일부 문자열이 깨져 보인다. 이는 파일 인코딩이 혼재되었을 가능성이 높다.

영향:

- 유지보수성 저하
- 일부 UI 문자열 품질 저하 가능

권장:

- 전체 소스를 UTF-8로 통일
- 깨진 주석/문자열 정비

## 14.2 민감 정보 관리 리스크

- `server/serviceAccount.json` 존재
- 직원 개인정보 필드(`rrn`, 주소, 전화번호 등) 직접 저장

권장:

- 서비스 계정 키를 환경변수/비밀 저장소로 이전
- 주민등록번호 등 민감정보는 저장 최소화 또는 암호화 검토

## 14.3 로직 중복

비밀번호 초기화가 다음 경로에 중복 존재한다.

- Express Admin API
- Firebase Callable Function
- 로그인 모달 내 자동 계정 생성/초기화 보조 흐름

운영 표준 경로를 하나로 정리하는 것이 좋다.

## 14.4 미사용/구버전 흔적 가능성

- `server/db.ts` 의 시드 함수 미호출 가능성
- `client/src/services/leaveService.ts` 는 현재 화면 흐름에서 직접 사용 흔적이 약함
- `server/` 와 `functions/` 가 비슷한 책임을 일부 공유

## 14.5 공지 읽음 처리 미완성 가능성

`Notice.readBy` 필드는 존재하지만 현재 확인한 코드에서는 읽음 상태를 저장하는 업데이트 로직이 없다.

## 15. 권장 개선 방향

### 단기

- 소스 인코딩 UTF-8 통일
- 문자열 상수/상태값 enum 정리
- Firestore 컬렉션/필드 명세를 별도 타입 파일로 통합

- 페이지별 Firestore 접근 로직을 서비스 레이어로 분리

## 17. 결론

이 프로젝트는 Firebase를 중심으로 빠르게 구축된 실무형 HR 관리 시스템이다. 화면 수와 기능 범위가 넓고, 실시간 조회와 관리자 중심 운영 기능이 강하다. 반면 데이터 접근이 프런트엔드에 많이 분산되어 있고, 인코딩/중복 책임/민감정보 관리 측면에서 정리가 필요한 상태다.

향후 유지보수의 핵심은 다음 세 가지다.

- 권한/인증 흐름 표준화
- 데이터 모델과 서비스 계층 정리
- 운영 보안 및 인코딩 품질 개선
